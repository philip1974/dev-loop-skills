# /req-doc — 生成专业需求文档（参考驱动 · 抗漂移）

把「我想要某功能 + 我参考了哪些 GitHub 项目」变成一份**AI 不敢漂移**的需求文档。**只产出文档**：不建 topic、不跑 red-team、不动项目代码。产物可直接喂给 `/dl-req` 或人读。

## 不要在这些时候用

- 用户只是闲聊功能点子 / 问问题。
- 用户要的是完整 dev-loop 立项追踪（用 `/dl-req`）。
- 用户要的是直接动手实现。

## 为什么这套不漂（核心原则）

1. **锚定 ≠ 描述。** 漂移发生在"用户转述 → AI 再解释"这层。所以每条需求**指向具体物**（repo + 路径/函数/行为），不让 AI 有转述空间。
2. **只装三件承重**（codex 四轮裁剪结论）：**功能取舍**（管范围）/ **验收锚点**（把"像参考"变成能失败的检查）/ **必须背离参考**（防奴性模仿）。其余字段都是这三者的附属，按需才填。
3. **按「承诺面」分档，不按感觉。** 写不写数据 / 暴不暴露 API / 影不影响用户路径 = 客观信号，决定该用便签还是契约。
4. **默认便签。** 宁可轻了再加，不可一上来镶大理石边。

## 流程

### Phase A — 定档（按承诺面）

按下表判断，**默认走最轻的一档**：

| 档 | 触发（满足即升档） | 用什么 |
|---|---|---|
| **便签** | <半天 · 无持久数据 · 无权限/计费 · 只改 UI 或局部交互 | 4 节：参考 / 要什么 / 不要什么 / 验收 |
| **精简契约** | >2 天 · 或 写数据 / 暴露 API / 影响用户路径 / 引入依赖 | 三件承重 + 瘦支撑 |
| **全套** | 多参考 + 多接缝 + 上线后难回滚 | 精简契约 + 全部瘦支撑展开 |

把判定结果和理由告诉用户，让其确认或改档。

### Phase B — 轻量问答（每轮 2-3 问，逐项抽锚点）

**不要问"你想要什么样的 X"（散文题），要逼用户指具体物。** 按档位提问：

1. **意图 + 参考**："要做什么功能？参考了哪些 GitHub 项目——给 repo 链接 + 具体到哪个文件/功能/行为（不是项目名）。"
2. **功能取舍**（每个参考逐功能）："这个参考，你**要它哪些**功能（adopt）、**改造哪些**（adapt：借语义不照搬实现）、**不要哪些**（reject）？" —— `reject/不要` 这半是最关键的负向锚。
3. **验收**："完成后怎么算'像参考'？给 1-3 条 given→期望，锚到参考的**实际行为**；每条说**怎么验**（命令/截图对比/fixture/人工看）。"
4. **必须背离**："什么情况下**必须不像**这个参考？（它的做法接不进你的系统 / 你的约束不同 / 有更适合的解）"
5. （仅当 Phase A 命中外部承诺）**集成风险**："碰了 数据 / 权限 / API / 钱 / 持久化 里的哪几样？各带进什么你不想要的假设？"

不替用户回答；用户说"不知道/你定"就记成开放项，不编。

### Phase C — 写需求文档

按档位用下面模板写。**没内容的节直接删，不留空表格。**

### Phase D — 交付

1. 问用户写到哪（默认 `./需求-<slug>.md` 或用户指定路径）。
2. 写文件。
3. 报告：路径 + 一句话"下一步可 `/dl-req` 立项，或直接照此实现"。**不自动做下一步。**

## 模板

### 便签档

```markdown
# 需求便签：<标题>

- **参考**：<repo@commit 链接> — 看的是这个版本的 <具体路径/功能>
- **要什么**：<adopt 的功能/行为>
- **不要什么**：<reject —— 这个参考里别学的部分>
- **验收**：<given → 期望；怎么验>
```

### 精简契约档

```markdown
# 需求文档：<标题>
> 契约：AI 必须严格遵循以下条目。**碰 MUST / MUST-NOT 必须停下问，不得静默偏离**（静默偏离 = 违约）。

## 参考清单
- [R1] <repo>@<commit>（<license>）— 采用理由（一行）

## 功能取舍（管范围）
| ID | 功能 | 来源 | adopt | adapt（借语义不照搬） | reject（MUST NOT） |
|---|---|---|---|---|---|
| F1 | ... | R1:<路径> | ... | ... | ... |

## 行为说明（behavior_notes，按需）
- F1：<意图 / 关键状态 / 边界与反例，写在一处>

## 验收锚点（管验收 · 每条带验证法）
| ID | given → 期望 | 锚到参考哪段行为 | verify_method |
|---|---|---|---|
| A1 | ... | R1:<路径/行为> | 命令 / 截图对比 / fixture / 人工审 |
> 没有 verify_method 的条目**不能当硬约束**，标"人工审查"。模糊语义（"气质像 X"）走人工审查门：差异表 + 截图/录屏 + 偏差解释，由用户确认才算过。

## 必须背离参考（管方向盘）
- <什么情况下必须不像参考——它为它的语境设计，照搬进你的语境本身就是失败模式>

## 集成风险（仅碰外部承诺时填：数据 / 权限 / API / 钱 / 持久化）
- <带进的隐藏假设 + 你认哪些 / 防火墙隔哪些 / 拒哪些>
```

全套档 = 精简契约 + 把 behavior_notes、集成风险逐项展开 + 参考清单补 license/活跃度（仅当要长期引入依赖或 fork 时）。

## 硬规则（双向防）

- **防漂移**：每条需求锚到具体 repo+路径，不接受纯散文；验收必须可对照（机器或人工审查门）；契约档必含"碰 MUST/MUST-NOT 必须停"。
- **防过度工程**：默认便签；没内容的节删掉不留空表；不强制每条写证据链（保留 ID 即可）；不默认开 `boundary/集成风险`，只在真碰外部承诺时填。
- **只写文档**：不建 dev-loop topic、不调用 codex、不改项目代码、不自动进下一阶段。
- **不替用户拍板**：参考选型、取舍、背离规则都由用户定；AI 只抽锚点、补结构、指出缺口。
